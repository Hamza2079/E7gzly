## Overview
**E7gzly** is a real-time, online queue management platform designed for medical clinics. It bridges the gap between doctors trying to manage crowded waiting rooms and patients wanting to skip physical wait times. 

The application is split primarily into two massive domains:
- **The Patient Experience**: A seamless, consumer-facing application for browsing doctors, joining queues remotely, managing favorite providers, and leaving clinic reviews.
- **The Provider Dashboard**: A powerful CRM and queue control center where doctors can actively serve patients, manage professional clinic profiles, view revenue reports, and respond to feedback.

---

## Tech Stack
- **Framework**: Next.js 14+ (App Router focused)
- **Language**: TypeScript
- **Styling**: Tailwind CSS, Lucide React (Icons)
- **Database / Backend**: Supabase (PostgreSQL)
- **Authentication**: Supabase Auth (Email/Pass & OAuth integration)
- **Form Handling**: Next.js Native Server Actions
- **Security**: PostgreSQL Row Level Security (RLS) & Next.js Server-Side auth checks

---

## Architecture & Patterns
- **Server Components By Default**: E7gzly relies heavily on React Server Components (RSC) to directly query the database via Supabase. This guarantees zero client-side waterfall fetching and enforces strict RLS policies securely on the backend.
- **Server Actions**: All mutations (Profile edits, Queue joining, Review responses) utilize `actions.ts` files containing `"use server"` declarations. Form submissions trigger server-side revalidations (`revalidatePath`), syncing UI instantly without heavy state management libraries.
- **Proxy Middleware Enforcement**: A centralized `proxy.ts` middleware system acts as the absolute gatekeeper, ensuring Doctors cannot access Patient-only routes (`/my-queue`) and Patients cannot access Dashboard routes (`/dashboard`).

## Role-Based Access Control (RBAC)

E7gzly controls access via a central `role` column on the `public.users` table (`patient`, `provider`, `admin`).
- **Patient**: Can read global doctor metrics, insert/cancel their own queuing records, update their own demographic forms, and leave reviews.
- **Provider (Doctor)**: Has unrestricted UPDATE and READ writes onto rows targeting their specific `provider.id` spanning schedules, reviews, providers matrix, and targeted queue sessions.
- **Admin**: God mode across the global schema metrics.

---

## Core Database Schema

The Postgres cluster rests on Supabase utilizing migration `.sql` scripts.
### Principal Tables
- **`users`**: The central auth extension. Holds full names, phones, and global application roles.
- **`providers`**: 1:1 linked to `users`. Holds consultation fees, bios, and specific clinic configurations.
- **`queues`**: Represents a specific working day session for a specific provider.
- **`queue_entries`**: The actual physical ticket generated by a patient when joining a line.
- **`reviews`**: Holds rating feedback mapping a patient to a provider regarding a specific queue entry.
- **`patient_favorites`**: An associative table allowing patients to bookmark providers.

*(All tables enforce PostgreSQL Row Level Security guaranteeing isolated horizontal tenancy without complex application layers).*

## Getting Started

### 1. Requirements
Ensure you are connected to the Supabase infrastructure. If changes were made to tables directly via migrations (`supabase/migrations/`), execute them against your Supabase project instance.

### 2. Environment Variables
Add your `.env.local` to the root folder:
```text
NEXT_PUBLIC_SUPABASE_URL="your-project-url"
NEXT_PUBLIC_SUPABASE_ANON_KEY="your-anon-key"
```

### 3. Running Locally
```bash
npm install
npm run dev
```

### 4. Updating Typescript Schema (Optional)
If database tables change shapes in the Supabase Dashboard, align the frontend interface typings using:
```bash
npx supabase gen types typescript --linked > types/database.types.ts
```
